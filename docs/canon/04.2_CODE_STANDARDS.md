# Purpose

This document establishes the permanent coding standards for EZ. It
ensures every line of code is readable, consistent, maintainable, and
production-ready regardless of who writes it.

------------------------------------------------------------------------

# Guiding Principles

Engineering should optimize for:

-   Readability over cleverness
-   Consistency over personal preference
-   Simplicity over complexity
-   Long-term maintainability over short-term speed

Code is written for humans first and computers second.

------------------------------------------------------------------------

# General Standards

Every contribution should be:

-   Small and focused
-   Strongly typed
-   Self-explanatory where possible
-   Free from unnecessary duplication
-   Consistent with the existing architecture

------------------------------------------------------------------------

# Naming Conventions

## Files

-   Use descriptive names.
-   React components: `PascalCase`
-   Utilities: `camelCase`
-   Constants: `UPPER_SNAKE_CASE` only when appropriate.

## Components

Component names should clearly describe their responsibility.

Examples:

-   `JobCard`
-   `ApplicationTimeline`
-   `InterviewScheduler`

Avoid vague names like:

-   `Card2`
-   `Helper`
-   `Manager`

------------------------------------------------------------------------

# Functions

Function names should describe actions.

Good examples:

-   `createApplication()`
-   `calculateMatchScore()`
-   `sendInterviewReminder()`

Avoid generic names such as:

-   `doStuff()`
-   `handleThing()`
-   `processData()`

------------------------------------------------------------------------

# Variables

Prefer descriptive names over abbreviations.

Good:

-   `applicationStatus`
-   `interviewDate`
-   `matchingScore`

Avoid:

-   `a`
-   `tmp`
-   `val`

except for short-lived loop variables.

------------------------------------------------------------------------

# TypeScript Standards

-   Prefer explicit types.
-   Avoid `any`.
-   Reuse shared interfaces.
-   Create domain-specific types when appropriate.
-   Keep types close to the feature unless shared globally.

------------------------------------------------------------------------

# React Standards

-   Prefer functional components.
-   Keep components focused on a single responsibility.
-   Extract reusable logic into hooks.
-   Avoid deeply nested component trees.

------------------------------------------------------------------------

# Imports

Import order:

1.  External libraries
2.  Internal shared modules
3.  Feature modules
4.  Relative imports

Remove unused imports before merging.

------------------------------------------------------------------------

# Comments

Code should be self-documenting whenever possible.

Use comments to explain **why**, not **what**.

Avoid redundant comments.

------------------------------------------------------------------------

# Error Handling

-   Handle expected failures gracefully.
-   Never silently ignore errors.
-   Show user-friendly messages.
-   Log diagnostic information for developers.

------------------------------------------------------------------------

# Clean Code Checklist

Before merging, verify:

-   Meaningful names
-   No dead code
-   No duplicated logic
-   Small functions
-   Clear responsibilities
-   Consistent formatting
-   No unnecessary complexity

------------------------------------------------------------------------

# Definition of Production-Ready Code

Production-ready code must:

-   Compile successfully
-   Pass tests
-   Follow the Design Canon
-   Respect the Experience Canon
-   Align with the Product Philosophy
-   Follow this Engineering Canon

------------------------------------------------------------------------

# Governance

Every pull request should answer:

> Does this code improve the product while remaining simple, readable,
> and maintainable?

If not, revise before merging.
