# Purpose

This document defines the permanent state management architecture for
EZ. It establishes where application state lives, who owns it, and how
it flows throughout the product to ensure consistency, scalability, and
reliability.

------------------------------------------------------------------------

# State Management Philosophy

State should exist in only one place.

Every piece of state must have:

-   A clear owner
-   A single source of truth
-   A predictable lifecycle
-   A defined scope

Duplicate state introduces bugs and inconsistency.

------------------------------------------------------------------------

# Core Principles

-   Keep state as local as possible.
-   Promote state only when multiple consumers require it.
-   Derive values instead of duplicating them.
-   Synchronize state through well-defined interfaces.
-   Remove unused state promptly.

------------------------------------------------------------------------

# State Hierarchy

    Server State
          ↓
    Global Application State
          ↓
    Feature State
          ↓
    Component State

Each level should depend only on the layer above when necessary.

------------------------------------------------------------------------

# Types of State

## Component State

Temporary UI state that belongs to a single component.

Examples:

-   Modal visibility
-   Input values
-   Dropdown selection
-   Hover state

Keep this local.

------------------------------------------------------------------------

## Feature State

State shared within a single feature.

Examples:

-   Job application workflow
-   Resume builder progress
-   Interview scheduling flow

Feature state should not leak into unrelated modules.

------------------------------------------------------------------------

## Global Application State

Shared application-wide data.

Examples:

-   Authenticated user
-   User preferences
-   Theme configuration
-   Notification settings

Global state should remain small and intentional.

------------------------------------------------------------------------

## Server State

Data owned by backend services.

Examples:

-   Jobs
-   Applications
-   Messages
-   User profile

Never duplicate server state unnecessarily.

------------------------------------------------------------------------

## AI Conversation State

AI interactions require dedicated state.

Track:

-   Conversation history
-   Context window
-   User intent
-   Pending AI actions
-   AI-generated drafts

AI state should remain isolated from unrelated application state.

------------------------------------------------------------------------

# State Ownership

Every state object must have exactly one owner.

Other modules consume state---they do not own it.

Ownership should always be explicit.

------------------------------------------------------------------------

# Data Flow

Recommended flow:

    User Action
          ↓
    State Update
          ↓
    Business Logic
          ↓
    UI Refresh

Data should flow in one predictable direction.

------------------------------------------------------------------------

# Derived State

Whenever possible:

Compute values instead of storing duplicates.

Examples:

-   Progress percentage
-   Completed applications
-   Match score summaries

Derived state reduces synchronization errors.

------------------------------------------------------------------------

# Caching

Cache only when it provides measurable value.

Cache should:

-   Expire appropriately
-   Be invalidated predictably
-   Never become the source of truth

------------------------------------------------------------------------

# Persistence

Persist only meaningful user data.

Examples:

-   Preferences
-   Drafts
-   Saved searches

Temporary UI state should not be persisted.

------------------------------------------------------------------------

# Performance

Avoid:

-   Excessive global state
-   Unnecessary updates
-   Deep state nesting
-   Repeated calculations

Optimize only after measuring real performance issues.

------------------------------------------------------------------------

# Error Recovery

State transitions should:

-   Recover gracefully
-   Prevent corruption
-   Preserve user work whenever possible

------------------------------------------------------------------------

# Testing

State logic should be independently testable.

Validate:

-   Initialization
-   Updates
-   Edge cases
-   Recovery
-   Synchronization

------------------------------------------------------------------------

# Governance

Before introducing new state, ask:

-   Does this already exist?
-   Can it be derived?
-   Who owns it?
-   What is its lifecycle?
-   Does it need to be shared?

If ownership is unclear, redesign before implementation.
