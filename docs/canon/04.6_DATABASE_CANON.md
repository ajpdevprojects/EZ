# Purpose

This document defines the permanent database architecture and data
governance standards for EZ. It ensures that all data is modeled
consistently, remains secure, and supports long-term scalability.

------------------------------------------------------------------------

# Database Philosophy

The database is the single source of truth.

Engineering should prioritize:

-   Accuracy
-   Integrity
-   Security
-   Performance
-   Scalability
-   Maintainability

Data models should reflect the business domain, not implementation
shortcuts.

------------------------------------------------------------------------

# Core Principles

-   Model real-world concepts clearly.
-   Store each fact only once.
-   Favor explicit relationships.
-   Keep schemas understandable.
-   Evolve through controlled migrations.

------------------------------------------------------------------------

# Naming Standards

## Tables

-   Singular or plural naming must remain consistent across the entire
    project.
-   Use lowercase with snake_case.

Examples:

-   users
-   jobs
-   applications
-   interviews

## Columns

Use descriptive snake_case names.

Examples:

-   created_at
-   updated_at
-   application_status
-   interview_date

Primary keys should use `id`.

Foreign keys should clearly identify the related entity.

------------------------------------------------------------------------

# Relationships

Prefer explicit foreign keys.

Relationships should:

-   Enforce referential integrity
-   Avoid unnecessary duplication
-   Support predictable querying

------------------------------------------------------------------------

# Normalization

Normalize by default.

Duplicate data only when there is a measured performance benefit and the
duplication can be maintained safely.

------------------------------------------------------------------------

# Indexing

Create indexes for:

-   Frequently queried fields
-   Foreign keys
-   Searchable attributes
-   Sorting fields

Review indexes periodically as the application evolves.

------------------------------------------------------------------------

# Migrations

All schema changes must occur through version-controlled migrations.

Migration rules:

-   Small and reversible
-   Tested before deployment
-   Never modify production data without a migration plan

------------------------------------------------------------------------

# Data Integrity

Protect integrity using:

-   Constraints
-   Foreign keys
-   Transactions
-   Validation
-   Consistent business rules

Invalid data should never become persistent.

------------------------------------------------------------------------

# Security

Database security must include:

-   Authentication
-   Authorization
-   Encryption where appropriate
-   Principle of least privilege
-   Secure credentials management

Sensitive information must never be exposed unnecessarily.

------------------------------------------------------------------------

# Row Level Security (RLS)

Where supported, enable Row Level Security.

Policies should ensure users can access only the data they are
authorized to view or modify.

------------------------------------------------------------------------

# Audit & Logging

Track important changes such as:

-   Record creation
-   Updates
-   Deletions
-   Permission-sensitive operations

Audit records should support debugging, compliance, and accountability.

------------------------------------------------------------------------

# Backup & Recovery

Maintain reliable backups.

Recovery plans should be documented and regularly tested to minimize
data loss.

------------------------------------------------------------------------

# Performance

Optimize through:

-   Efficient queries
-   Proper indexing
-   Pagination
-   Batching
-   Monitoring query performance

Avoid premature optimization.

------------------------------------------------------------------------

# Testing

Validate:

-   Migrations
-   Constraints
-   Relationships
-   RLS policies
-   Performance-critical queries
-   Backup and recovery procedures

------------------------------------------------------------------------

# Governance

Before changing the schema, ask:

-   Does this represent the business domain correctly?
-   Can existing structures be reused?
-   Will this remain understandable in the future?
-   Does it preserve data integrity and security?

If not, redesign before implementation.
