# Purpose

This document defines the permanent repository architecture for EZ. It
establishes how the codebase is organized so every contributor and AI
agent builds consistently.

The repository must be:

-   Predictable
-   Modular
-   Scalable
-   Maintainable
-   Easy to navigate

------------------------------------------------------------------------

# Engineering Philosophy

The repository is a product.

A clean repository reduces bugs, speeds development, simplifies
onboarding, and enables long-term maintainability.

Every directory should have a single responsibility.

------------------------------------------------------------------------

# Architectural Principles

## Separation of Concerns

Business logic, presentation, infrastructure, AI, data access, and
utilities must remain separated.

## Feature First

Features own their implementation while sharing common infrastructure.

## Reuse Before Reinvention

Shared components, utilities, and services should be reused before
creating new implementations.

## Single Source of Truth

Business rules must exist in only one location.

------------------------------------------------------------------------

# Recommended Top-Level Structure

    app/
    components/
    features/
    lib/
    services/
    hooks/
    types/
    styles/
    public/
    docs/
    tests/
    scripts/

Each directory has a clear purpose and should not duplicate
responsibilities.

------------------------------------------------------------------------

# Directory Responsibilities

### app/

Application routes, layouts, and page entry points.

### components/

Reusable UI components that are shared across features.

### features/

Feature-specific business logic, screens, and workflows.

### lib/

Core utilities, helpers, configuration, and shared infrastructure.

### services/

External integrations, APIs, AI providers, and backend communication.

### hooks/

Reusable custom hooks.

### types/

Shared TypeScript types and interfaces.

### styles/

Global styles, design tokens, and theme definitions.

### tests/

Unit, integration, and end-to-end tests.

### docs/

Canonical documentation and engineering references.

------------------------------------------------------------------------

# Dependency Rules

Dependencies should always flow inward.

    Pages
        ↓
    Features
        ↓
    Components
        ↓
    Services / Lib

Lower layers must never depend on higher layers.

------------------------------------------------------------------------

# Shared Code Rules

Promote code to shared modules only after it has proven reusable.

Avoid premature abstraction.

------------------------------------------------------------------------

# Naming Conventions

-   One concept, one name.
-   Descriptive directory names.
-   Avoid ambiguous abbreviations.
-   Keep file names consistent throughout the repository.

------------------------------------------------------------------------

# Documentation

Every major module should include:

-   Purpose
-   Responsibilities
-   Dependencies
-   Usage guidelines

------------------------------------------------------------------------

# Repository Governance

Before adding a new directory, ask:

-   Does a suitable location already exist?
-   Can this extend an existing feature?
-   Does it improve long-term maintainability?

If the answer is no, redesign before implementation.

------------------------------------------------------------------------

# Repository Freeze

This architecture is the permanent organizational foundation of EZ.

Changes require Founder approval to preserve consistency.
